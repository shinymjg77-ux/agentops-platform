# 4단계 계획(Phase 4): 운영 고도화 / 외부 공개 준비

## 1. 단계 목표
- 인증/권한 체계를 도입해 다중 사용자 운영 기반을 완성한다.
- 외부 접속 가능한 HTTPS 배포 기준을 확립한다.
- 실운영 알림 채널과 백업/복구 체계를 정착한다.

## 2. 완료 기준(Definition of Done)
- 로그인/권한 기반 접근 제어가 API/대시보드에 적용된다.
- 외부 도메인 + TLS 환경에서 정상 접속된다.
- Slack/Telegram 중 1개 이상 정책 알림이 실제 수신된다.
- 백업 실행/복구 리허설 기록이 남고 성공한다.
- 감사 로그 화면에서 핵심 이벤트 조회가 가능하다.
- Phase 4 E2E/운영 체크리스트가 통과한다.

## 3. 작업 백로그
1. 사용자/프로젝트/RBAC 데이터 모델 추가
2. 인증 API(`login/logout/me`) 구현
3. API 보호 미들웨어(JWT/세션) 적용
4. 대시보드 로그인 및 권한별 메뉴 제어
5. 프로젝트 스코프(리소스 필터링) 반영
6. 알림 채널 모델/API + 정책 채널 바인딩
7. Slack/Telegram 알림 어댑터 구현
8. Reverse proxy + HTTPS 구성 문서/템플릿
9. 백업 실행 스크립트 + 이력 저장(`backup_runs`)
10. 복구 리허설 절차/체크리스트 문서화
11. 감사 로그 수집 API/화면 구현
12. Phase 4 E2E + 운영 점검 스크립트 추가
13. Runbook/PRD/세션 로그 갱신

## 4. 권장 일정(8~10영업일)
- Day 1: 데이터 모델 + 인증 API 골격
- Day 2: API 보호 미들웨어 + 기존 엔드포인트 점진 적용
- Day 3: 대시보드 로그인/권한 UI
- Day 4: 프로젝트 스코프 반영
- Day 5: 알림 채널/정책 바인딩
- Day 6: Slack/Telegram 실제 발송 검증
- Day 7: HTTPS 배포 템플릿 및 보안 점검
- Day 8: 백업/복구 기능 + 감사 로그
- Day 9~10: E2E/문서화/안정화

## 6. 우선순위
- P0:
  - 인증/권한
  - HTTPS 외부 접속
  - 알림 채널 1개 이상 실운영 검증
- P1:
  - 백업/복구 자동화
  - 감사 로그 UI
- P2:
  - 다중 채널 정책, 고급 노이즈 억제

## 7. 테스트 전략
- 단위:
  - 권한 매트릭스(역할별 허용/거부)
  - 토큰 만료/재로그인 플로우
- 통합:
  - 프로젝트 스코프 데이터 격리 검증
  - 정책 알림 발송 성공/실패 경로 검증
- E2E:
  - 로그인 → 작업 조회 → 권한 제한 확인
  - 정책 트리거 → 채널 수신 확인
  - 백업 실행 → 이력 기록 확인

## 8. 리스크 및 완화
- 기존 기능 회귀:
  - Phase1~3 E2E를 회귀 세트로 항상 병행
- 운영 복잡도 증가:
  - runbook 우선 작성, 설정 템플릿 제공
- 인증 전환 시 사용자 혼선:
  - 마이그레이션 공지 + 단계별 롤아웃

## 9. 의존성/사전 준비
- 도메인/인증서 준비(또는 테스트 도메인)
- 알림 채널 토큰/웹훅(비밀값 관리 정책 포함)
- 백업 스토리지 경로/보관 정책 합의
