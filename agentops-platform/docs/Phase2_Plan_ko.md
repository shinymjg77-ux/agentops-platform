# 2단계 계획(Phase 2): 템플릿 레지스트리 + 버전 비교 + 고급 검색

## 1. 단계 목표
- 템플릿을 코드 하드코딩이 아닌 레지스트리로 관리한다.
- 실행 이력에 템플릿/어댑터 버전을 기록하고 비교 가능하게 만든다.
- 운영자가 기간/상태/템플릿/오류 기준으로 이력을 탐색할 수 있게 한다.

## 2. 완료 기준(Definition of Done)
- 템플릿 레지스트리 CRUD API/화면이 동작한다.
- 작업 실행 시 `template_version`(필수), `adapter_version`(선택)이 run 이력에 저장된다.
- 버전 비교 화면에서 템플릿 버전별 성공률/실패율/평균시간을 확인할 수 있다.
- 고급 검색 화면에서 필터 조합(기간/상태/템플릿/키워드) 조회가 가능하다.
- Phase 2 E2E 테스트(정상/검색/비교)가 자동화되어 통과한다.

## 4. 작업 백로그
1. DB 스키마 확장(`task_templates`, `task_template_versions`, run 메타컬럼)
2. 기존 샘플 템플릿을 레지스트리 데이터로 이관
3. 템플릿 레지스트리 조회/등록/수정 API
4. 템플릿 버전 활성화/비활성화 및 기본버전 API
5. 작업 생성 시 `template_name` 기반 레지스트리 조회 + 버전 resolve
6. `task_runs`에 `template_version`, `adapter_version`, `error_code` 기록
7. 이력 검색 API(필터/페이지네이션/정렬)
8. 버전 비교 API(집계: 성공률/실패율/평균시간)
9. 대시보드: 템플릿 레지스트리 목록/상세 UI
10. 대시보드: 버전 비교 UI
11. 대시보드: 고급 검색 UI
12. API/프론트 통합 E2E 테스트
13. 문서/런북 업데이트

## 5. 권장 일정(5~7영업일)
- Day 1: DB 확장 + 레지스트리 API 초안
- Day 2: 작업 생성/실행 경로에 버전 메타 연결
- Day 3: 검색/비교 API 구현
- Day 4: 레지스트리/검색/비교 UI 구현
- Day 5: E2E 테스트 + 안정화
- Day 6~7(버퍼): 마이그레이션/성능 점검/문서 보강

## 6. 리스크 및 대응
- 기존 하드코딩 템플릿과 레지스트리 간 불일치
  - 대응: 초기 시드 데이터 + fallback 조회 + 검증 로그
- 검색 쿼리 성능 저하
  - 대응: 인덱스 설계(`status`, `template_id`, `started_at`) + limit/offset 제한
- 버전 비교 지표 해석 혼선
  - 대응: 비교 기준(기간/샘플수 최소조건)을 UI에 명시
